# Current Operations Checklist

## Purpose

This file provides a practical checklist for using the Service Availability Matrix in current operations.

It exists to make the matrix more actionable before full automation is built.

This is the operational checklist layer of the tenant matrix.

## Why This Exists

The matrix is already strong as documentation and governance logic.

But operationally, the platform still needs a simple human-usable checklist for:

- reviewing current tenants
- spotting weak areas
- deciding what to strengthen next
- checking whether new changes affect readiness, parity, or profile fit

This file makes the matrix usable in current manual operations.

## Core Principle

Before automation, there should be disciplined manual use.

A simple repeatable checklist is one of the fastest ways to move from passive documentation to operational habit.

## Checklist A — Baseline Business Check

For each tenant, confirm:

- tenant README exists
- business README exists
- business-model README exists
- canvas overview exists
- communications README exists
- strategy README exists
- one short-form pitch exists
- one deeper audience-facing communication exists
- tenant matrix entry exists

If any baseline item is missing:
- record it
- classify whether this blocks readiness
- decide whether correction is needed now or later

## Checklist B — Profile Fit Check

For each tenant, confirm:

- strongest profile is identified
- the tenant still fits that profile
- profile-specific gaps are visible
- profile-specific differences are not being mistaken for defects

If profile fit feels weak or unclear:
- record it
- consider profile reassessment
- do not force false parity too early

## Checklist C — Surface Check

For each tenant, confirm:

- current surface expectations are visible
- website/UI/dashboard state is known
- role-based or stakeholder/service-module surfaces are understood
- absence is explained where needed

If surface state is unclear:
- record it
- improve documentation
- avoid assuming capabilities that are not visible

## Checklist D — Bundle Check

For each tenant, confirm:

- expected bundles are visible
- bundle incompleteness is known
- partial bundles are not being mistaken for full bundles
- strong bundle patterns are being noticed for future reuse

If a bundle is incomplete:
- decide whether this is normal
- decide whether this is rollout
- decide whether this is under-provisioning

## Checklist E — Promotion Check

At each review, ask:

- did any repeated tenant pattern become stronger
- is any pattern now reusable
- is any pattern ready for standardization
- is any pattern ready to become a registration default

If yes:
- add or update promotion candidates
- update standardization thinking
- consider governance review if needed

## Checklist F — Parity Check

At each review, ask:

- is shared baseline still aligned
- are profile differences still legitimate
- is any difference now looking accidental
- did a rollout change make a previous difference no longer acceptable

If yes:
- choose parity action
- decide escalation level if needed

## Suggested Review Rhythm

Use this checklist:

- after meaningful tenant changes
- after new matrix files are added
- after new business/service structures are created
- before stronger registration or provisioning changes
- at major server-10 checkpoints

## Current Use

At the current stage, this checklist should be treated as:

- manual
- lightweight
- repeatable
- governance-aware

It is a bridge between documents and future automation.

## Final Principle

A matrix becomes operational when it changes review behavior.

This checklist exists to make that happen now, even before automation is built.
