# Provisioning Gap Summary

## Purpose

This file summarizes the current gap between what the Service Availability Matrix already knows and what current tenant provisioning still does not yet operationalize.

It exists so the platform can see how far it is from profile-aware, matrix-informed onboarding.

This is the provisioning-gap layer of the tenant matrix.

## Why This Exists

The matrix now has enough logic to describe:

- shared baseline
- profile fit
- bundle expectations
- readiness thresholds
- parity interpretation

But provisioning is not yet fully using that logic.

This file makes that gap visible.

## Current Strong Matrix Knowledge

The matrix already knows with fairly strong confidence:

- what the business minimum baseline is
- what communication minimums are emerging
- which tenant best fits role-surface pattern
- which tenant best fits stakeholder/service-module pattern
- what basic registration-ready structure looks like

These are real strengths.

## Current Provisioning Gaps

### Gap 1 — Baseline Is Not Yet Fully Provisioned by Rule

Current problem:
- the platform knows the baseline, but future tenant creation is not yet fully driven by that baseline

What is missing:
- baseline-aware scaffolding logic
- baseline-aware onboarding sequence execution
- default creation of baseline matrix files

## Gap 2 — Profiles Are Defined but Not Yet Provisioning-Active

Current problem:
- profiles exist, but they do not yet actively shape provisioning outcomes

What is missing:
- role-surface profile scaffold logic
- stakeholder/service-module profile scaffold logic
- hybrid profile scaffold logic

## Gap 3 — Bundle Logic Is Visible but Not Yet Provisioning-Linked

Current problem:
- bundles are interpreted, but not yet used to provision grouped expectations

What is missing:
- bundle-based starting structure
- bundle-aware checklists
- bundle-aware surface expectations

## Gap 4 — Matrix Enrollment Is Still Mostly Manual

Current problem:
- tenant matrix files exist, but they are not yet automatically or consistently tied to onboarding flow

What is missing:
- onboarding step that always creates tenant matrix entry
- onboarding step that updates summaries intentionally

## Gap 5 — Readiness Gates Are Not Yet Provisioning Gates

Current problem:
- readiness is documented, but does not yet influence how provisioning stops, pauses, or warns

What is missing:
- baseline failure warnings
- partial onboarding flags
- registration-ready review checkpoints

## Current Honest State

Provisioning is currently:
- conceptually informed by matrix logic
- not yet strongly driven by matrix logic

This means the platform has stronger governance understanding than provisioning execution.

## Best Near-Term Provisioning Improvements

The most useful near-term improvements are likely:

- baseline-aware scaffolding
- automatic tenant matrix file creation
- profile-choice awareness during onboarding
- onboarding checklist use during tenant creation
- clearer post-onboarding readiness review

## Final Principle

The matrix is already teaching the platform how provisioning should improve.

This file exists so that teaching does not remain invisible.
