# Automation Candidate Summary

## Purpose

This file summarizes which current Service Availability Matrix judgments are strongest candidates for simple automation.

It exists so the platform can move from human-only review toward selective low-risk automation.

This is the automation-candidate layer of the tenant matrix.

## Why This Exists

The matrix is now broad and mature enough that some parts are stronger candidates for automation than others.

The platform should start with what is:

- clear
- low-ambiguity
- high-signal
- easy to verify

This file identifies those candidates.

## Strongest Automation Candidates

### 1. Baseline File Presence Checks

Examples:
- tenant README exists
- business README exists
- business-model README exists
- canvas overview exists
- communications README exists
- strategy README exists

Why strong:
- direct evidence
- low ambiguity
- high operational value

## 2. Communication Minimum Presence Checks

Examples:
- short-form pitch exists
- deeper audience-facing communication exists

Why strong:
- stable pattern
- easy to verify
- useful for readiness summary

## 3. Tenant Matrix Enrollment Checks

Examples:
- tenant matrix file exists
- tenant appears in summary layers if expected

Why strong:
- simple to check
- high governance value

## 4. Shared Baseline Satisfaction Summary

Examples:
- tenant satisfies all baseline items
- tenant partially satisfies baseline
- tenant fails baseline

Why strong:
- directly derived from file presence
- useful for onboarding and readiness

## Moderate Automation Candidates

### Profile-Supporting Surface Hints
Examples:
- user-interface presence
- website presence
- obvious customer/employee folders
- obvious stakeholder/service-module areas

Why moderate:
- visible enough to help
- but still somewhat interpretive

### Bundle Presence Hints
Examples:
- several expected bundle elements are present
- several expected bundle elements are missing

Why moderate:
- useful for hints
- but weaker than strict baseline checks

## Weaker Automation Candidates

These should remain mostly manual for longer:

- final profile assignment
- hybrid profile judgment
- promotion threshold crossing
- exception validity
- strong parity enforcement
- standard-review decisions

Why weaker:
- more nuanced
- more interpretive
- higher risk of false precision

## Current Recommended Automation Order

### First
- baseline file presence
- communication minimum presence
- tenant matrix entry presence

### Second
- readiness summary hints
- surface-presence hints
- bundle incompleteness hints

### Later
- profile-aware warnings
- promotion-candidate hints
- stronger provisioning linkage

## Final Principle

Automation should begin where the matrix is clearest.

This file exists so early automation starts with low-risk, high-value checks instead of trying to automate everything at once.
