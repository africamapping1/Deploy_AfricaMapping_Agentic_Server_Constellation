# Tenant Matrix Limitations Summary

## Purpose

This file summarizes the current most important limitations affecting tenant use of the Service Availability Matrix.

It exists so the platform can see what still constrains matrix maturity in practice.

This is the limitations-summary layer of the tenant matrix.

## Main Current Limitations

### 1. Deeper Business-Model Detail Is Still Weak
Both current tenants still rely mostly on business-model anchors rather than deeper structural component files.

### 2. Richer Communications Maturity Is Still Early
Both current tenants still lack some richer communication layers such as:
- positioning
- operator explanation
- objection handling

### 3. Provisioning Is Not Yet Strongly Matrix-Driven
The matrix teaches provisioning clearly, but provisioning is not yet strongly using that knowledge.

### 4. Automation Is Still Narrow and Future-Facing
Automation targets are identified, but most matrix use is still human-guided.

### 5. Profile Logic Is Stronger in Documentation Than in Live Enforcement
Profiles are well-described, but not yet heavily used in live warnings or control logic.

## Current Interpretation

These limitations do not undermine the matrix.

They mainly show that the matrix is strongest in:
- governance
- comparison
- readiness
- onboarding guidance

and still earlier in:
- operational behavior
- automation
- enforcement

## Final Principle

Limitations are not failures.

They are the remaining space between current maturity and future operational maturity.

This file exists so that space stays visible.
