# First Socket Candidates

## Purpose

This file identifies the strongest first candidates for backend socketing inside the Service Availability Matrix.

It exists so the platform can start with the highest-value and lowest-ambiguity connections.

This is the first-socket-candidates file of the tenant matrix.

## Why This Exists

The matrix is broad enough that not all socketing should happen at once.

The first sockets should be practical, interpretable, and low risk.

## Strongest First Candidates

### 1. Baseline Presence Checks
Examples:
- required minimum files exist
- tenant matrix entry exists
- communication minimum exists

Why strong:
- low ambiguity
- easy to review
- good first automation candidate later

### 2. Review Checklist Use
Examples:
- onboarding checklist
- manual review checklist
- promotion review checklist

Why strong:
- converts matrix into habit
- helps test practical usefulness immediately

### 3. Summary Freshness Checks
Examples:
- summary files updated after meaningful structural changes
- stale summaries flagged for review

Why strong:
- supports trust
- avoids silent drift

### 4. Profile-Aware Onboarding Guidance
Examples:
- different scaffolding hints for role-surface versus stakeholder/service-module tenants

Why strong:
- directly useful
- helps future tenant setup

### 5. Promotion Readiness Packaging
Examples:
- structured readiness summary prepared for governance review

Why strong:
- creates a real bridge from matrix language to approval behavior

## Important Note

The first socket candidates should prefer:

- low-risk use
- high clarity
- repeated usefulness
- limited noise

## Final Principle

The earliest sockets should prove value before larger automation is attempted.

This file exists so the platform begins from the strongest candidates first.
