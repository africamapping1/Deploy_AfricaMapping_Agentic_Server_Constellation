# Profile Drift Watchlist

## Purpose

This file tracks where tenant profile fit may be changing over time or may deserve closer review.

It exists so the platform can notice profile evolution before it becomes confusing parity or onboarding logic.

This is the profile-drift watchlist of the tenant matrix.

## Why This Exists

A tenant profile is not always permanent.

A tenant may begin as:

- role-surface leaning
- stakeholder/service-module leaning

and later evolve.

Without a watchlist, profile shifts may be missed until parity or provisioning logic becomes confusing.

## Core Principle

Profile drift is not automatically a problem.

It is a signal that the tenant may be evolving into a different or richer pattern.

## Watchlist Triggers

A tenant should be reviewed again for profile drift when:

- new role-based surfaces are added
- new stakeholder/service modules are added
- bundle interpretation becomes more mixed
- communications become broader across profile boundaries
- parity rules become harder to interpret cleanly

## Final Principle

A tenant profile should not be frozen too early.

This file exists so the platform can watch profile movement without overreacting to healthy growth.
