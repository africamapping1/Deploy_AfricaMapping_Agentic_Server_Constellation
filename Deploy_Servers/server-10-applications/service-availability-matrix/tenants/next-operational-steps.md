# Next Operational Steps

## Purpose

This file lists the most practical near-term steps for making the Service Availability Matrix more operational.

It exists so the platform can move from rich documentation toward repeatable use.

This is the next-operations layer of the tenant matrix.

## Why This Exists

The matrix is now strong enough that the biggest remaining gap is less about understanding and more about usage.

This file keeps the next operational moves simple and visible.

## Near-Term Steps

### 1. Fix Any Matrix Hygiene Issues
Examples:
- file name typos
- missing summary updates
- missing tenant references

### 2. Start Using the Onboarding Checklist
Use the onboarding checklist whenever a new tenant or major tenant branch is created.

### 3. Start Using the Current Operations Checklist
Use it after meaningful tenant changes.

### 4. Begin Baseline Presence Checking
Even manually at first, regularly confirm:
- baseline files exist
- communication minimum exists
- tenant matrix entry exists

### 5. Keep Summary Files Updated
After meaningful tenant or layer changes, update the affected summaries.

### 6. Preserve Change Log Discipline
Record meaningful interpretation changes rather than relying on memory.

### 7. Prepare the First Small Automation Pass
Aim first at:
- baseline file presence
- communication minimum presence
- matrix entry presence

### 8. Keep Profiles Realistic
Avoid over-forcing hybrid or advanced assumptions until real evidence is stronger.

## Final Principle

The next operational step is not to automate everything.

It is to use the matrix regularly enough that it becomes part of how the platform thinks and acts.

This file exists so that step remains practical.
