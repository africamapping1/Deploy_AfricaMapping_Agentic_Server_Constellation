# Decision Output Types

## Purpose

This layer defines the kinds of decisions the Service Availability Matrix should help produce.

It exists so matrix review can move toward clearer outputs instead of vague observations.

This is the decision-output layer of the Service Availability Matrix.

## Why This Exists

A matrix is most useful when it helps the platform reach decisions.

Without named output types, review may end in:
- interesting notes
- partial conclusions
- unclear next steps

This layer makes outputs clearer.

## Core Principle

A matrix review should end in one or more clear output types.

## Suggested Output Types

### 1. Baseline Decision
Examples:
- baseline satisfied
- baseline partial
- baseline failed

### 2. Readiness Decision
Examples:
- not ready
- baseline ready
- registration-ready
- registration-ready with notes

### 3. Profile Decision
Examples:
- role-surface
- stakeholder/service-module
- hybrid
- unclear for now

### 4. Parity Decision
Examples:
- aligned
- profile-aligned
- conditionally aligned
- under-provisioned
- drift detected

### 5. Action Decision
Examples:
- no action
- advisory strengthening
- parity correction
- profile reassessment
- promotion review

### 6. Promotion Decision
Examples:
- keep under observation
- promotion candidate
- ready for standard review
- not promotable yet

## Why This Helps

Decision outputs improve:
- review consistency
- change logging
- governance clarity
- future automation suitability

## Final Principle

A matrix becomes more operational when it produces decisions, not only descriptions.

This layer exists so those decision types are visible and reusable.
