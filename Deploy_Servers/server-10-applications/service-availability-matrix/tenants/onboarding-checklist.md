# Tenant Onboarding Checklist

## Purpose

This file provides a practical onboarding checklist for shaping a new tenant using the Service Availability Matrix.

It exists so the platform can begin using matrix logic operationally before full automation exists.

This is the onboarding checklist layer of the tenant matrix.

## Why This Exists

The platform now has:

- registration baseline logic
- profile logic
- bundle logic
- readiness logic
- parity logic

What is still useful is a simple sequence that a human can follow while creating a new tenant.

This file provides that sequence.

## Step 1 — Create Tenant Identity

Confirm:
- tenant root README created
- tenant business directory created
- tenant business README created

Goal:
- the tenant exists as a recognizable business/application line

## Step 2 — Create Minimum Business Foundation

Confirm:
- business-model README created
- canvas overview created
- communications README created
- strategy README created

Goal:
- the tenant is structurally readable as a business

## Step 3 — Create Minimum Communication Layer

Confirm:
- short-form pitch created
- one deeper audience-facing communication created

Goal:
- the tenant can explain itself at more than one depth

## Step 4 — Enroll the Tenant in the Matrix

Confirm:
- tenant matrix file created
- readiness summary updated if needed
- profile-fit interpretation recorded
- summary layers updated if needed

Goal:
- the tenant becomes visible to governance and parity

## Step 5 — Assign a Registration Profile

Confirm:
- role-surface, stakeholder/service-module, hybrid, or other profile chosen
- profile rationale noted
- unclear profile status explicitly documented if needed

Goal:
- the tenant is not compared blindly against the wrong expectations

## Step 6 — Clarify Surface Expectations

Confirm:
- website expectation known
- UI expectation known
- dashboard expectation known
- customer/employee/stakeholder/operator surface expectations known
- optional or not-applicable states noted where relevant

Goal:
- the tenant has a visible service/surface shape

## Step 7 — Identify Bundle Expectations

Confirm:
- baseline bundle known
- profile-specific bundle expectation known
- optional or delayed bundle items noted

Goal:
- the tenant’s capability grouping is visible, not implicit

## Step 8 — Review Readiness

Confirm:
- baseline ready or not
- parity-visible or not
- registration-ready or not
- key gaps recorded

Goal:
- the tenant leaves onboarding with a visible maturity judgment

## Step 9 — Record Next Strengthening Priorities

Confirm:
- first major next gaps recorded
- first likely maturity improvements identified
- parity-sensitive risks noted if any

Goal:
- onboarding ends with direction, not only structure

## Final Principle

A new tenant should not be created as a pile of folders.

It should be onboarded through a governed sequence.

This checklist exists to make that process repeatable before automation exists.
