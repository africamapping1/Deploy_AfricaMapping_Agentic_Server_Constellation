# Tenant Bundle Summary

## Purpose

This file provides a comparative summary of capability bundles across tenants.

It exists so the platform can see not only which individual services are present, but which groups of capabilities are already emerging as meaningful tenant bundles.

This is the bundle comparison layer above individual tenant matrix files.

## Why This Exists

Single-item tracking is useful, but it can become too fragmented.

The platform also needs to understand grouped value such as:

- business minimum bundle
- role-surface bundle
- stakeholder/service-module bundle
- integration-oriented bundle

This file helps the platform compare those bundle patterns across tenants.

## Current Compared Tenants

At this time, the bundle summary compares:

- `africamapping`
- `realestate-ai`

Future tenants should be added here as they are created.

## Shared Learning

The platform is already learning that not every tenant should be treated as the same bundle profile.

At least three bundle patterns are emerging:

- common business minimum bundle
- role-surface bundle
- stakeholder/service-module bundle

This is an important step toward smarter tenant registration and parity checks.

## Reusability Judgment

### Strongest Shared Bundle
- business minimum bundle

### Strongest Role-Oriented Bundle
- role-surface bundle, currently modeled most strongly by `realestate-ai`

### Strongest Stakeholder / Module-Oriented Bundle
- stakeholder/service-module bundle, currently modeled most strongly by `africamapping`

## Current Gaps

The platform still lacks:

- explicit bundle definitions as governed provisioning units
- bundle-level rollout logic
- bundle-to-tenant-type mapping
- bundle-aware parity checks
- bundle-aware registration defaults

These are future strengthening areas.

## Suggested Next Moves

- define bundle states more explicitly
- map bundle expectations to tenant types
- define which bundles are default, optional, or conditional
- connect bundles to future registration and provisioning logic
- later connect bundles to doctor or parity-check automation

## Final Interpretation

The platform is now mature enough to see that tenants are not only collections of files.

They are beginning to express recurring capability bundles.

This means future growth can be guided by bundle patterns instead of isolated manual additions.
