# Next Milestones

## Purpose

This layer defines the next meaningful milestones for maturing the Service Availability Matrix.

It exists so the platform can move through visible milestones instead of vague ongoing expansion.

This is the milestone layer of the Service Availability Matrix.

## Why This Exists

The matrix is now broad enough that future progress should be milestone-based.

This helps the platform know when it has:
- finished a documentation phase
- crossed into stronger review use
- crossed into early automation
- crossed into stronger operational control

## Suggested Near-Term Milestones

### Milestone 1 — Matrix Hygiene Stabilized
Success means:
- naming issues fixed
- summaries synchronized
- key missing files filled
- matrix tree trustworthy

### Milestone 2 — Manual Review Habit Established
Success means:
- onboarding checklist is actually used
- current operations checklist is actually used
- change log discipline starts being real

### Milestone 3 — First Automation Pass Exists
Success means:
- baseline presence checks run
- communication minimum checks run
- matrix file presence checks run
- output is low-noise and readable

### Milestone 4 — Provisioning Influence Begins
Success means:
- onboarding sequence affects real scaffolding
- baseline files are created more consistently
- tenant matrix creation becomes part of onboarding

### Milestone 5 — Doctor Visibility Begins
Success means:
- low-risk matrix signals begin appearing in platform health visibility
- baseline weakness becomes visible beyond docs alone

## Suggested Later Milestones

### Milestone 6 — Profile-Aware Advisory Warnings
### Milestone 7 — Bundle-Aware Provisioning Guidance

## Final Principle

Milestones turn matrix maturity into something measurable.

This layer exists so progress can be tracked in meaningful chunks.
