# Profile Defaults

## Purpose

This layer defines the default expectations that should normally accompany each registration profile.

It exists so the platform can provision more intelligently once a profile is chosen.

This is the profile-defaults layer of the Service Availability Matrix.

## Why This Exists

Profiles are useful only if they affect real expectation.

Once a tenant is understood as:

- role-surface
- stakeholder / service-module
- hybrid

the platform should know what normally comes with that shape.

This layer makes those defaults explicit.

## Core Principle

A profile should imply default expectations, not rigid identity.

Defaults can guide onboarding while still allowing justified exceptions.

## Shared Defaults for All Profiles

Every profile should normally include:

- tenant README
- business README
- business-model README
- canvas overview
- communications README
- strategy README
- one short-form pitch
- one deeper audience-facing communication
- tenant matrix entry

These are shared defaults.

## Role-Surface Profile Defaults

Typical defaults may include:

- user-interface presence
- customer-facing visibility expectation
- employee-facing visibility expectation
- dashboard expectation
- role-aware surface interpretation
- operator-sensitive communications growth later

These are defaults, not yet universal hard requirements.

## Stakeholder / Service-Module Profile Defaults

Typical defaults may include:

- stakeholder-facing communication expectation
- operations area expectation
- broader service-module grouping
- partner or institutional communication growth later
- business-area readability emphasis

These are defaults, not rigid mandates.

## Hybrid Profile Defaults

Typical defaults may include:

- shared baseline
- role-aware surface expectation
- stakeholder-aware communication expectation
- broader combined service logic
- richer communication growth expectation

Because hybrid is still earlier, these defaults should remain softer for now.

## Why Defaults Matter

Defaults help the platform avoid:
- forgetting obvious expected structures
- forcing every tenant into the same onboarding shape
- leaving profile logic too abstract to be useful

## Relationship to Exceptions

Defaults are not absolute.

A tenant may differ if the difference is:
- visible
- justified
- profile-aware
- reviewed where needed

## Final Principle

A profile becomes more operational when it carries defaults.

This layer exists so profile knowledge becomes practical without becoming rigid.
