# Tenant Lifecycle

## Purpose

This layer defines how the Service Availability Matrix should understand tenants as moving entities rather than fixed folder states.

It exists so the matrix can track tenant growth over time.

This is the tenant-lifecycle layer of the Service Availability Matrix.

## Why This Exists

A tenant does not stay in one maturity state forever.

A tenant may move through stages such as:

- initial scaffold
- baseline shaping
- registration readiness
- bundle growth
- profile deepening
- parity stabilization
- future restructuring

Without a lifecycle view, the matrix may treat every tenant as if it should already be complete.

This layer provides a more realistic growth model.

## Core Principle

A tenant should be understood as evolving through stages.

The matrix should describe that evolution, not erase it.

## Suggested Lifecycle Stages

### 1. Scaffold Stage

The tenant exists only in early structural form.

Typical signs:
- root folders exist
- README may be missing or weak
- business meaning still shallow

### 2. Foundation Stage

The tenant begins to show minimum business structure.

Typical signs:
- tenant README
- business README
- business-model README
- canvas overview
- communications README
- strategy README

### 3. Communication Stage

The tenant becomes able to explain itself more clearly.

Typical signs:
- short-form pitch
- deeper audience-facing pitch
- stronger communications interpretation

### 4. Matrix Stage

The tenant becomes visible to matrix-based governance.

Typical signs:
- tenant matrix entry
- readiness summary
- profile fit visibility
- parity visibility

### 5. Registration-Ready Stage

The tenant becomes strong enough to act as a proper governed instance.

Typical signs:
- baseline passed
- profile understood
- surfaces known
- parity comparable

### 6. Bundle Growth Stage

The tenant begins to express stronger bundle patterns.

Typical signs:
- role-surface pattern
- stakeholder/service-module pattern
- integration grouping
- stronger capability clustering

### 7. Standard Influence Stage

The tenant begins to teach the platform reusable patterns.

Typical signs:
- promotion candidates
- standardization influence
- registration baseline influence
- profile refinement influence

## Why Lifecycle Matters

Lifecycle thinking helps the platform ask better questions:

- what stage is this tenant in
- what is appropriate now
- what is premature now
- what is the next strengthening step
- should this tenant be compared to mature tenants yet

## Relationship to Readiness

Readiness is a threshold judgment.

Lifecycle is a growth path.

They work together:
- lifecycle explains where the tenant is coming from
- readiness explains what threshold it has crossed

## Relationship to Parity

Parity should be aware of lifecycle stage.

A younger tenant may deserve different expectations than a tenant already influencing platform standardization.

This prevents unfair comparison.

## Final Principle

A tenant is not only a current state.

It is a moving business line.

This layer exists so the matrix can govern tenant growth with more realism and discipline.
