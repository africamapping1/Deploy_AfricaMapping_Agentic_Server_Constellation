# Readiness Gates

## Purpose

This layer defines the gates a tenant should pass through as it moves from basic existence toward stronger onboarding, parity participation, and future provisioning confidence.

It exists so readiness is not treated as one vague state.

This is the readiness-gate layer of the Service Availability Matrix.

## Why This Exists

A tenant can exist in the filesystem and still be weak in business meaning.

A tenant can also have baseline files and still be weak in:

- communications
- surface clarity
- profile fit
- matrix visibility
- onboarding strength

This means readiness should be gated in stages.

## Core Principle

A tenant should move through visible readiness gates.

Each gate should represent a stronger level of structural and operational trust.

# Readiness Gates

## Purpose

This layer defines the gates a tenant should pass through as it moves from basic existence toward stronger onboarding, parity participation, and future provisioning confidence.

It exists so readiness is not treated as one vague state.

This is the readiness-gate layer of the Service Availability Matrix.

## Why This Exists

A tenant can exist in the filesystem and still be weak in business meaning.

A tenant can also have baseline files and still be weak in:

- communications
- surface clarity
- profile fit
- matrix visibility
- onboarding strength

This means readiness should be gated in stages.

## Core Principle

A tenant should move through visible readiness gates.

Each gate should represent a stronger level of structural and operational trust.

## Suggested Gates

### Gate 1 — Identity Gate

The tenant must have:

- tenant README
- business README

This means the tenant exists as a recognizable business/application line.

### Gate 2 — Business Foundation Gate

The tenant must have:

- business-model README
- canvas overview
- communications README
- strategy README

This means the tenant has minimum business readability.

### Gate 3 — Communication Gate

The tenant must have:

- one short-form pitch
- one deeper audience-facing communication

This means the tenant can explain itself at more than one depth.

### Gate 4 — Matrix Visibility Gate

The tenant must have:

- tenant matrix entry
- visible readiness interpretation
- visible profile fit interpretation

This means the tenant is no longer invisible to parity and review logic.

### Gate 5 — Surface Clarity Gate

The tenant must have:

- explicit known surfaces
- known missing surfaces where relevant
- profile-aware surface interpretation

This means the tenant is no longer only described, but meaningfully surfaced.

### Gate 6 — Registration Readiness Gate

The tenant must be judged:

- baseline ready
- parity-visible
- profile-understood
- acceptable for governed onboarding reference

This means the tenant can now serve as a valid registered business instance.

## Gate States

Useful gate states may include:

- not passed
- partially passed
- passed
- passed with notes

These help the platform express nuance without losing direction.

## Why Gates Matter

Gates help the platform avoid mistakes such as:

- treating a half-shaped tenant like a mature tenant
- comparing parity too early
- onboarding from weak examples
- promoting patterns that have not crossed enough structure

## Relationship to Enforcement

Readiness gates may later influence:

- informational notes
- warning thresholds
- gated review
- future onboarding blocks

But at the current stage they should mainly guide disciplined review.

## Final Principle

A tenant should not become trusted all at once.

This layer exists so trust grows through visible gates rather than assumption.
