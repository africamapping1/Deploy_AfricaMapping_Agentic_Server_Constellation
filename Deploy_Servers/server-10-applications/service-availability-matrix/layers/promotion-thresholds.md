# Promotion Thresholds

## Purpose

This layer defines how the platform should decide when a tenant-specific pattern is strong enough to be promoted upward into reusable shared structure, standard bundle status, or registration default logic.

It exists so promotion is not done by intuition alone.

This is the promotion-decision layer of the Service Availability Matrix.

## Why This Exists

The platform is already beginning to observe repeated patterns across tenants, such as:

- common business minimum structures
- communication baselines
- role-surface patterns
- stakeholder/service-module patterns

But not every repeated pattern should be promoted immediately.

A pattern may be:

- too weak
- too tenant-specific
- too early
- too uneven
- not yet stable enough

This layer exists so the platform can judge promotion carefully and consistently.

## Core Principle

Repetition alone is not enough.

A pattern should be promoted only when it is sufficiently:

- present
- comparable
- mature
- useful
- repeatable
- stable

Promotion should be earned by evidence.

## What Promotion Means

A pattern may be promoted into one of several higher states:

### 1. Shared Business Framework Promotion

Use when the pattern is reusable business intelligence.

Examples:

- communication structures
- canvas structures
- perspective logic
- business-model logic

### 2. Application-Plane Standard Promotion

Use when the pattern is becoming part of the expected architecture of tenants.

Examples:

- standard tenant business minimums
- standard communications baseline
- standard strategy presence
- standard matrix enrollment expectations

### 3. Capability Bundle Promotion

Use when related items belong together and should be assigned as a governed grouped package.

Examples:

- business minimum bundle
- role-surface bundle
- stakeholder/service-module bundle

### 4. Registration Default Promotion

Use when the pattern is strong enough to be provisioned automatically or semi-automatically during future tenant onboarding.

## Promotion Threshold Questions

When considering promotion, the platform should ask:

### Presence Threshold
- does the pattern appear in more than one tenant
- is it more than accidental repetition

### Comparability Threshold
- are the tenant versions structurally comparable
- do they solve a similar problem in a similar way

### Maturity Threshold
- are the examples documented, usable, and clear
- or are they still weak placeholders

### Stability Threshold
- has the pattern remained useful across changes
- or is it still experimental

### Utility Threshold
- would promotion make the platform stronger
- would future tenants benefit from it
- would parity become clearer because of it

### Boundary Threshold
- does the pattern belong in tenant space
- or is it truly shared enough to move upward

## Suggested Promotion States

A pattern under consideration may be described as:

- not promotable
- under observation
- promotion candidate
- ready for shared framework promotion
- ready for standard bundle promotion
- ready for registration default promotion

These states help the platform distinguish between observation and action.

## Minimum Promotion Logic

A pattern should usually not be promoted unless it satisfies most of the following:

- appears across multiple tenants
- is structurally comparable
- is more than scaffolded
- solves a repeated need
- improves parity or onboarding clarity
- does not depend too heavily on one tenant’s unique context

This is not a rigid formula, but it is a useful discipline.

## Examples of Likely Promotion Candidates

### Strong Candidate
- tenant business minimum baseline

Why:
- appears across current tenants
- structurally comparable
- already useful for registration thinking
- clearly improves onboarding clarity

### Moderate Candidate
- one short-form pitch baseline

Why:
- repeated across tenants
- useful for communication maturity
- likely relevant to future tenants

### Tenant-Type Candidate
- role-surface bundle

Why:
- clearly strong in one tenant
- visible enough to compare
- likely useful, but not yet universal

### Tenant-Type Candidate
- stakeholder/service-module bundle

Why:
- clearly strong in one tenant
- visible enough to compare
- likely useful, but not yet universal

### Weak or Early Candidate
- integration-oriented bundle

Why:
- pattern exists
- but still too uneven to promote strongly yet

## Relationship to Tenant Types

Promotion should sometimes happen at the tenant-type level instead of the universal level.

That means a pattern may be ready for promotion as:

- a role-surface tenant standard
- a stakeholder/service-module tenant standard

without yet becoming a universal default for all tenants.

This is an important distinction.

## Relationship to Parity

Promotion thresholds help parity because they answer:

- what should now be expected
- what is still optional
- what is still emerging
- what absence is acceptable
- what absence is now a drift concern

This keeps parity from becoming arbitrary.

## Relationship to Future Registration

Promotion thresholds are especially important for future onboarding.

A pattern should not become registration default too early.

But once it crosses the threshold, it should no longer depend on memory.

This means promotion thresholds help convert learned practice into governed onboarding logic.

## Governance Value

This layer helps the platform avoid two opposite mistakes:

- promoting weak patterns too early
- failing to promote strong patterns when they are clearly ready

It makes upward movement more deliberate and more legible.

## Final Principle

A mature platform should know not only what is repeated.

It should know when repetition has become evidence.

This layer exists so platform promotion decisions are based on visible thresholds, not only intuition.
