# Socketable Subsection Types

## Purpose

This layer identifies the major kinds of Service Availability Matrix subsections that are expected to receive backend sockets.

It exists so socketing work can begin from recognizable matrix surfaces.

This is the socketable-subsections layer of the Service Availability Matrix.

## Why This Exists

Not all matrix content is equally operational.

Some parts are especially suited for backend connection.

## Major Socketable Subsection Types

### 1. Baseline and Minimum Layers
These define:

- what must exist
- what must be present
- what counts as minimum acceptable tenant structure

These are strong candidates for:
- provisioning sockets
- checklist sockets
- automation sockets later

### 2. Readiness and Promotion Layers
These define:

- what counts as mature enough
- what gates progression
- what conditions support promotion

These are strong candidates for:
- governance sockets
- review sockets
- approval sockets

### 3. Parity and Drift Layers
These define:

- shared expectation
- acceptable difference
- baseline drift
- misalignment
- comparability limits

These are strong candidates for:
- monitoring sockets
- summary maintenance
- doctor visibility
- warning behavior later

### 4. Profile Layers
These define:

- what kind of tenant or business surface is being represented
- what strength pattern applies
- how onboarding expectations should differ by profile

These are strong candidates for:
- provisioning sockets
- onboarding sockets
- governance interpretation

### 5. Review and Checklist Layers
These define:

- what humans or agents should inspect
- what should be asked
- what should be confirmed before progression

These are strong candidates for:
- manual review habit
- agent-assisted review
- governance packaging

### 6. Summary and Stop-Point Layers
These define:

- current interpretation
- maturity assessment
- practical value
- next priority direction

These are strong candidates for:
- monitoring visibility
- review packaging
- executive signal generation

## Final Principle

Socketing should begin where matrix content is already closest to action.

This layer exists so that operational connection starts from the strongest surfaces first.
