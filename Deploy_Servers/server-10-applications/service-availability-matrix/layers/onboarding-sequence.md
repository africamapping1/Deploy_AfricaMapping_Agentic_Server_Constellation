# Onboarding Sequence

## Purpose

This layer defines the recommended order in which a new tenant should be shaped inside server-10-applications.

It exists so onboarding follows a governed sequence rather than a random collection of folders and files.

This is the onboarding-sequence layer of the Service Availability Matrix.

## Why This Exists

A tenant is easier to build well when the order is disciplined.

Without sequence, the platform may create:

- surfaces before business truth
- communications before structure
- parity comparisons before readiness
- bundles before profile fit

This layer exists so tenant onboarding grows in a logical order.

## Core Principle

A tenant should be built from identity toward business truth, then toward communication, then toward surfaces, then toward parity and operational control.

## Recommended Sequence

### Step 1 — Tenant Identity
Create:
- tenant root README
- tenant business directory
- tenant business README

This establishes that the tenant exists as a business/application line.

### Step 2 — Business Structure
Create:
- business-model README
- canvas overview
- strategy README
- communications README

This establishes minimum business readability.

### Step 3 — Communication Baseline
Create:
- one short-form pitch
- one deeper audience-facing communication

This establishes the minimum communication baseline.

### Step 4 — Matrix Enrollment
Create:
- tenant matrix file
- readiness visibility
- baseline interpretation

This makes the tenant visible to parity and review logic.

### Step 5 — Profile Fit
Determine:
- role-surface
- stakeholder/service-module
- hybrid
- or other future profile

This prevents later parity confusion.

### Step 6 — Surface Expectations
Define:
- website
- UI
- dashboards
- role surfaces
- stakeholder/service modules
- optional integrations

This makes service presence explicit.

### Step 7 — Bundle Interpretation
Determine:
- which bundle patterns apply
- which are partial
- which are optional
- which are future rollout items

### Step 8 — Readiness Review
Evaluate:
- baseline ready
- parity-ready
- registration-ready
- still partial

## Why This Order Matters

This sequence keeps the platform from:
- communicating before it understands
- exposing surfaces before it defines purpose
- comparing parity before profile fit exists
- assuming readiness before structure is real

## Relationship to Registration Profiles

A registration profile should influence the sequence, but not erase it.

Profiles help shape what gets emphasized.

The sequence still keeps onboarding ordered.

## Relationship to Provisioning

Future provisioning should ideally follow this same sequence, even if it becomes automated.

That means onboarding automation should still respect the logic of staged tenant formation.

## Final Principle

A tenant should not appear fully formed by accident.

This layer exists so onboarding follows a governed path from identity to readiness.
