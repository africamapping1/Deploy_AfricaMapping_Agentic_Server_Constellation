# Confidence Levels

## Purpose

This layer defines how confidently the platform should hold different Service Availability Matrix judgments.

It exists so the matrix can distinguish between:

- strong conclusions
- useful working interpretations
- early hypotheses
- future-facing concepts

This is the confidence layer of the Service Availability Matrix.

## Why This Exists

Not every matrix judgment is equally certain.

For example:

- a missing README can be known with high confidence
- a tenant’s full hybrid profile fit may be much less certain
- a promotion candidate may be plausible but not yet fully proven

Without confidence levels, the matrix risks sounding equally certain about everything.

This layer exists to add judgment discipline.

## Core Principle

The platform should be explicit about how strongly it believes a matrix interpretation.

Confidence should reflect evidence quality, comparability, and stability.

## Suggested Confidence Levels

### 1. Low Confidence

Use when:
- evidence is weak
- the pattern appears only once
- the logic is still mostly theoretical
- the interpretation is early or speculative

Examples:
- early hybrid expectations
- emerging integration-oriented bundle standardization

### 2. Moderate Confidence

Use when:
- evidence is visible
- the pattern is meaningful
- but cross-tenant support is still incomplete
- or the logic is still evolving

Examples:
- role-surface bundle as reusable candidate
- stakeholder/service-module bundle as reusable candidate

### 3. Strong Confidence

Use when:
- evidence is direct and clear
- the pattern appears across tenants
- the interpretation is already useful in governance or onboarding thinking

Examples:
- tenant business minimum baseline
- short-form pitch baseline
- registration readiness for current modeled tenants

### 4. Very Strong Confidence

Use when:
- evidence is direct
- repeated
- structurally comparable
- stable over time
- already shaping standardization

Examples:
- future candidate: fully stabilized registration baseline
- future candidate: long-proven common bundle

## What Affects Confidence

Confidence should increase when:

- evidence is direct
- multiple tenants support the same interpretation
- content is strong, not merely present
- the pattern has survived review cycles
- the interpretation has real operational usefulness

Confidence should decrease when:

- the pattern appears only once
- the pattern is heavily profile-specific
- the logic is still highly interpretive
- rollout state is still unstable
- exceptions are too common or unclear

## Relationship to Promotion

Promotion decisions should be sensitive to confidence.

A low-confidence pattern should rarely be promoted.

A strong-confidence pattern is much more eligible for:
- standardization
- bundle promotion
- registration default use

## Relationship to Enforcement

Confidence should also influence enforcement.

High-confidence signals are better candidates for:
- warnings
- gated review
- eventual blocking

Low-confidence signals should usually stay:
- informational
- observational
- human-guided

## Final Principle

A mature matrix should not only say what it thinks.

It should know how strongly it thinks it.

This layer exists so the platform’s judgments remain honest, disciplined, and governable.
