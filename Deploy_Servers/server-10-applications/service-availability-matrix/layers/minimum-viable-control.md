# Minimum Viable Control

## Purpose

This layer defines the smallest set of Service Availability Matrix behaviors that would qualify as a real operational control system, even before advanced automation exists.

It exists so the platform knows what “minimum real control” looks like.

This is the minimum-control layer of the Service Availability Matrix.

## Why This Exists

The matrix is already very strong in documentation and governance.

But operational control does not need to wait for full automation.

There is a minimum viable control level where the matrix already begins to shape behavior meaningfully.

## Core Principle

Minimum control exists when the matrix changes what the platform does, not only what it says.

## Minimum Viable Control Conditions

A practical minimum control state would include:

### 1. Baseline Review Discipline
Every new or meaningfully changed tenant is checked against the current shared baseline.

### 2. Matrix Enrollment Discipline
Every serious tenant gets a tenant matrix file and becomes visible to summary logic.

### 3. Profile Awareness
A tenant is not reviewed blindly; its likely profile is identified before parity assumptions harden.

### 4. Summary Maintenance
Meaningful changes trigger summary updates.

### 5. Next-Action Use
The platform actually uses the matrix to choose what to strengthen next.

## Why This Counts as Control

This already means the matrix is affecting:

- onboarding behavior
- review behavior
- comparison behavior
- governance memory
- strengthening priorities

That is already operational control, even if it is still mostly manual.

## What Is Beyond Minimum Viable Control

More advanced stages include:
- automation
- warnings
- doctor integration
- provisioning hooks in practice
- gated review behavior

These are important, but they are beyond the minimum.

## Final Principle

The matrix becomes operational before it becomes automated.

This layer exists so the platform can recognize that minimum real control threshold.
