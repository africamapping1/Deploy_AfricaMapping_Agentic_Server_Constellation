# Matrix Anti-Patterns

## Purpose

This layer defines the behaviors and mistakes the platform should avoid while using the Service Availability Matrix.

It exists so the matrix matures without becoming heavy, noisy, or misleading.

This is the anti-pattern layer of the Service Availability Matrix.

## Why This Exists

A strong matrix can still be used poorly.

This layer names common mistakes so the platform can avoid them early.

## Anti-Patterns

### 1. False Sameness

Mistake:
- assuming every tenant should look identical

Why harmful:
- ignores profile difference
- creates bad parity judgments
- weakens tenant-type thinking

## 2. Empty Documentation Expansion

Mistake:
- adding many matrix files without improving actual clarity or decision quality

Why harmful:
- makes the matrix larger but weaker
- creates maintenance burden without value

## 3. Early Over-Automation

Mistake:
- automating nuanced judgments before logic is stable

Why harmful:
- creates false warnings
- reduces trust
- hardens weak assumptions

## 4. Silent Standardization

Mistake:
- letting something become standard without visible promotion reasoning

Why harmful:
- weakens governance
- confuses onboarding
- hides architectural drift

## 5. Ignoring Exceptions

Mistake:
- treating every difference as a defect

Why harmful:
- punishes valid profile differences
- weakens flexibility
- creates false parity noise

## 6. Never Retiring Weak Ideas

Mistake:
- keeping every bundle, concept, or rule forever even when it no longer helps

Why harmful:
- clutters the matrix
- makes interpretation noisy
- weakens trust in active logic

## 7. Summary Drift

Mistake:
- updating layer logic but not updating summary files

Why harmful:
- makes summary views misleading
- weakens operational use
- breaks trust in quick reading

## Final Principle

A mature matrix is not only built well.

It is also protected from predictable misuse.

This layer exists so the platform can avoid becoming heavy, rigid, or noisy.
