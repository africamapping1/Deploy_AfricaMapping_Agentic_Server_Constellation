# Summary Minimums

## Purpose

This layer defines the minimum things each matrix summary should make clear.

It exists so summary files remain useful and consistent.

This is the summary-minimums layer of the Service Availability Matrix.

## Why This Exists

The matrix now contains many summaries.

A summary is most useful when it clearly states:
- what it is summarizing
- what is being compared
- what the key interpretation is
- what the main next implication is

Without minimum expectations, summaries may become uneven.

## Core Principle

Every summary should help the reader answer:
- what is the state
- what does it mean
- what should we watch or do next

## Minimum Summary Elements

Each summary should ideally include:

### 1. Purpose
What the summary is for.

### 2. Current Compared Scope
Which tenants, bundles, or layers are being summarized.

### 3. Main Interpretation
What the summary most strongly says.

### 4. Major Differences
What differs in meaningful ways.

### 5. Main Risk or Gap
What remains weaker or unfinished.

### 6. Next Use
How the summary should influence review, governance, or next steps.

## Relationship to Summary Hierarchy

Higher-level summaries may be shorter.

Lower-level summaries may be more detailed.

But both should still satisfy the minimum purpose of orienting the reader.

## Final Principle

A summary should summarize something that matters.

This layer exists so summary writing stays disciplined and useful.
