# Baseline Check Logic

## Purpose

This layer defines how the platform should evaluate whether a tenant satisfies the current shared business minimum baseline.

It exists so baseline judgment is done consistently.

This is the baseline-check layer of the Service Availability Matrix.

## Why This Exists

The platform now has a meaningful shared tenant baseline.

But if the baseline is not checked through a stable logic, different reviews may reach different conclusions.

This layer provides that logic.

## Core Principle

Baseline judgment should be simple, visible, and low-ambiguity.

The platform should know exactly what is being checked.

## Current Baseline Items

At this stage, the current shared baseline includes:

- tenant README
- business README
- business-model README
- canvas overview
- communications README
- strategy README
- one short-form pitch
- one deeper audience-facing communication
- tenant matrix entry

These are the strongest current baseline signals across tenants.

## Check Logic

A tenant should be evaluated against each baseline item individually.

For each item, ask:

- is it present
- is it more than empty
- is it meaningful enough to count as documented
- is absence explained if relevant

## Suggested Baseline Outcomes

### Baseline Failed
Use when multiple required baseline items are missing.

### Baseline Partial
Use when some baseline items exist but the tenant is still incomplete.

### Baseline Satisfied
Use when the tenant has the current required baseline in usable documented form.

### Baseline Strong
Use when the tenant not only satisfies the baseline, but expresses it clearly enough to support comparison and reuse.

## Evidence Preference

Baseline checks should prefer:

- direct file presence evidence
- direct content evidence where possible

This keeps baseline judgment grounded and easy to verify.

## Relationship to Readiness

Baseline satisfaction is not the same as full maturity.

It is the first meaningful threshold.

A tenant may satisfy the baseline and still be early in:

- business-model depth
- advanced communications
- bundle maturity
- automation readiness

## Relationship to Automation

This is one of the strongest early automation candidates in the matrix because it relies on:

- stable file expectations
- visible structure
- low-ambiguity evidence

## Final Principle

A good baseline should be easy to check and hard to misunderstand.

This layer exists so baseline trust becomes consistent and governable.
