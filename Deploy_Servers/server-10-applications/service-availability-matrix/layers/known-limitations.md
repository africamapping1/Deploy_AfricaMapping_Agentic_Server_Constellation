# Known Limitations

## Purpose

This layer defines the known current limitations of the Service Availability Matrix.

It exists so the platform remains honest about what the matrix can and cannot yet do.

This is the limitations layer of the Service Availability Matrix.

## Why This Exists

A strong matrix can still have real limits.

Naming those limits protects trust and helps focus improvement.

## Current Known Limitations

### 1. Limited Automation
Most matrix use is still manual or documentary.

### 2. Limited Doctor Integration
Matrix state is not yet strongly visible in system health reporting.

### 3. Limited Provisioning Influence
Provisioning is still not fully driven by matrix logic.

### 4. Limited Live Enforcement
Warnings, gates, and blocks are still mostly conceptual.

### 5. Limited Deep Business-Model Content
Tenants are strong in baseline anchors, but still shallow in deeper business-model detail.

### 6. Limited Advanced Communication Coverage
Tenants still lack richer maturity files such as:
- positioning
- operator explanation
- objection handling

### 7. Hybrid Profile Still Early
Hybrid remains useful conceptually, but still not strongly evidenced by a current tenant.

## Why Limitations Matter

Limitations do not make the matrix weak.

They make it realistic.

A realistic matrix is easier to improve and easier to trust.

## Final Principle

A mature platform should know not only its strengths, but also its current boundaries.

This layer exists so the matrix stays honest and focused.
