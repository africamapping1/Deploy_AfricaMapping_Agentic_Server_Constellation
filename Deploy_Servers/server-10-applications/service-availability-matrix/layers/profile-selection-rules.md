# Profile Selection Rules

## Purpose

This layer defines how the platform should choose an initial registration profile for a new or evolving tenant.

It exists so profile selection is deliberate and explainable.

This is the profile-selection layer of the Service Availability Matrix.

## Why This Exists

Profiles now matter for:
- onboarding
- parity
- bundle expectations
- readiness interpretation

That means profile selection should follow visible rules rather than guesswork.

## Core Principle

A profile should be selected from the tenant’s strongest visible operating pattern.

## Selection Rules

### Rule 1 — Start With Shared Baseline First
Do not choose a profile before confirming that the tenant is at least structurally visible enough to review.

### Rule 2 — Look for Strongest Surface Pattern
Ask:
- is the tenant strongest in role-aware user surfaces
- or strongest in stakeholder/service-module grouping
- or does it show both strongly

### Rule 3 — Look for Strongest Communication Pattern
Ask:
- does the tenant communicate more through role/customer/operator logic
- or more through stakeholder/partner/service logic

### Rule 4 — Look for Strongest Bundle Pattern
Ask:
- which bundle is most visible and mature
- which bundle best matches the tenant’s current shape

### Rule 5 — Choose the Simplest Strong Fit
If one profile clearly dominates, use that profile first.

Do not force hybrid too early unless both dimensions are strongly present.

## Suggested Selection Outcomes

- role-surface
- stakeholder / service-module
- hybrid
- unclear for now

## Relationship to Profile Drift

A selected profile is not permanent.

It should be reviewable as the tenant evolves.

## Final Principle

Choose the profile that best matches what is already visible.

This layer exists so profile selection stays evidence-based and calm.
