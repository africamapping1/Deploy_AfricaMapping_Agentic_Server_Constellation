# Hybrid Registration Profile

## Purpose

This file defines the hybrid registration profile for tenants whose business shape combines strong role-based user surfaces with strong stakeholder-facing or service-module structure.

It exists so the platform can support more complex tenants without forcing them into a single narrow pattern.

## Why This Exists

Some tenants will eventually need both of the following at once:

- role-aware user surfaces such as customer, employee, operator, or dashboard experiences
- broader stakeholder, partner, service-module, or integration-oriented business structure

These tenants should not be forced to choose one profile when their real business shape requires both.

This profile exists to support that combined reality.

## Core Principle

A hybrid tenant is a business that needs both:

- role-surface clarity
- stakeholder / service-module clarity

The platform should therefore provision both dimensions intentionally.

## Current Profile Meaning

At the current stage, the hybrid profile is more a forward-looking profile than a fully proven cross-tenant default.

It is useful because platform growth is likely to produce tenants that combine:

- dashboards and user roles
- stakeholder communication
- broader module groupings
- integration areas
- varied service surfaces

## Shared Baseline Included

This profile should always include the shared registration baseline:

- tenant README
- business README
- business-model README
- canvas overview
- communications README
- strategy README
- one short-form pitch
- one deeper audience-facing communication
- tenant matrix entry

## Hybrid Bundle Expectations

A hybrid tenant may combine elements such as:

- website or UI presence
- customer and employee surfaces
- dashboard expectations
- stakeholder-facing communication
- partner or institutional explanation
- operations structure
- service-module grouping
- optional integration-oriented business areas

## Communication Expectations

A hybrid tenant will usually need broader communication coverage than simpler tenant profiles.

Typical expectations may include:

- elevator pitch
- one deeper audience-facing pitch
- operator explanation
- stakeholder or partner explanation
- positioning
- future objection handling

The exact mix may still vary, but the communication layer should usually be richer than a minimal baseline.

## Business Model Emphasis

A hybrid tenant should usually strengthen business-model components such as:

- segments
- value proposition
- channels
- relationships
- key resources
- key activities
- key partners

This is because hybrid tenants often operate across multiple audiences and service modes.

## Surface Expectations

A hybrid tenant will often need a broader explicit surface profile, including some combination of:

- website
- UI
- dashboards
- customer surface
- employee surface
- operator surface
- stakeholder-facing visibility
- service-module grouping

Not all must be fully implemented immediately, but the profile should make them visible as expected or optional.

## Parity Interpretation

For this profile, parity should not ask only whether isolated files exist.

It should ask whether the tenant has the combined structure its profile expects.

Examples:

- are role-based surfaces defined
- are stakeholder/service structures defined
- are communication layers broad enough
- are omissions intentional or accidental

## Reusable Pattern Logic

The hybrid profile may later become important for more mature tenants that combine product, service, stakeholder, and integration complexity.

It may eventually support:

- more advanced onboarding profiles
- richer bundle combinations
- larger service availability expectations
- future premium or expanded tenant categories

## Current Caution

This profile should not yet be over-standardized.

It is still a useful conceptual and future-ready profile, but likely needs more real tenant evidence before becoming a stronger standard.

## Registration Starting State

A new tenant using this profile should ideally enter the matrix as:

- baseline ready
- hybrid bundle expected
- partial or registration-ready depending on how many combined surfaces are provisioned

## Final Principle

Some tenants will need more than one simple pattern.

This profile exists so the platform can handle richer tenant realities without collapsing them into the wrong starting structure.
