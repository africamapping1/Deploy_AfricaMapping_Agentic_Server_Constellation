# Role-Surface Registration Profile

## Purpose

This file defines the role-surface registration profile for tenants whose business shape depends strongly on distinct user roles, user-facing surfaces, and dashboard-oriented visibility.

It exists so the platform can onboard this kind of tenant from a governed profile instead of assembling the tenant structure manually each time.

## Why This Exists

Some tenants are shaped primarily around users and roles.

They depend on structures such as:

- customer-facing views
- employee-facing views
- role-aware communication
- dashboard visibility
- user-access mapping
- UI presence

These tenants should not be forced into a stakeholder/module-heavy pattern if that is not their dominant reality.

This profile exists to give them a more accurate starting shape.

## Core Principle

A role-surface tenant is a business whose operating clarity depends heavily on who is using the system and what each role sees.

The platform should therefore provision role-aware business and service structure from the start.

## Current Model Example

Current strongest example:
- `realestate-ai`

This tenant currently demonstrates the clearest role-surface pattern through customer-facing, employee-facing, dashboard, and UI-oriented structures.

## Shared Baseline Included

This profile should always include the shared registration baseline:

- tenant README
- business README
- business-model README
- canvas overview
- communications README
- strategy README
- one short-form pitch
- one deeper audience-facing communication
- tenant matrix entry

## Role-Surface Bundle Included

This profile should strongly expect a role-surface bundle including:

- user-interface presence
- customer surface expectation
- employee surface expectation
- dashboard expectation
- role-aware communication readiness
- user-access or role-mapping awareness

## Typical Surface Expectations

A tenant using this profile will often need:

- website or UI surface
- customer-facing view
- employee-facing view
- dashboards
- user or access structure
- operator explanation in communications
- role-sensitive service clarity

Not every element must be fully operational on day one.

But the profile should mark them as expected, optional, or pending.

## Communication Expectations

A role-surface tenant should usually include:

- elevator pitch
- one deeper audience-facing pitch
- future operator explanation
- future positioning
- future objection handling where commercially relevant

The deeper pitch type may vary, but role-aware communication should grow earlier in this profile than in some other tenant types.

## Business Model Emphasis

A role-surface tenant should usually strengthen business-model components such as:

- customer or user segments
- value proposition
- channels
- relationships
- key activities
- key resources

This is because role-based businesses often depend on clear differentiation between who uses the system and how value is experienced by different groups.

## Surface Status Guidance

Useful expected statuses for this profile may include:

- website: scaffolded or better
- user interface: scaffolded or better
- customer surface: documented or better
- employee surface: documented or better
- dashboards: documented, implemented, or operational
- access structure: documented or better

These are guidance signals, not rigid law.

## Registration Starting State

A new tenant using this profile should ideally enter the matrix as:

- baseline ready
- role-surface bundle expected
- registration-ready once minimum business and communication layers are present

## Parity Interpretation

For this profile, parity should not only check whether files exist.

It should ask whether the tenant has the expected role-aware structures for its class.

Examples:

- is customer visibility defined
- is employee visibility defined
- are dashboards expected and visible
- is the user-interface layer present
- is absence intentional or accidental

## Reusable Pattern Logic

This profile should help the platform reuse patterns such as:

- role-based business explanation
- customer and employee surface expectations
- dashboard-oriented tenant logic
- user-facing service packaging

These patterns may later become stronger provisioning defaults for future tenants of similar type.

## Current Limits

This profile should not yet assume that every role-surface tenant must be identical.

It should allow variation in:

- exact audience pitch type
- depth of customer versus employee features
- dashboard scope
- access model complexity

The profile is a starting pattern, not a final cage.

## Final Principle

A role-surface tenant should begin with role-aware structure, not discover it by accident later.

This profile exists so tenants of this kind start from a governed, comparable, and reproducible pattern.
